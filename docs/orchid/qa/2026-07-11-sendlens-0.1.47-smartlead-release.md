# SendLens 0.1.47 Smartlead release QA

Date: 2026-07-11

Issues: SENDOSS-97, SENDOSS-98

Parity PR: https://github.com/orchidautomation/sendlens-plugin/pull/51

Release branch: `codex/sendlens-0.1.47-smartlead-release`

## Release scope

- Merge the validated Smartlead parity work into its required `codex/smartlead-api-parity-map` base.
- Integrate that base with the current `main` release line without dropping the newer automatic-refresh, selector-error, host-runtime, or response-contract behavior.
- Publish Smartlead Standard API and support-gated Smart Delivery reads as SendLens V1 read-only surfaces.
- Rewrite the public README around a provider/capability matrix that distinguishes included, bounded, support-gated, and intentionally absent behavior.
- Bump the GitHub release version from `0.1.46` to `0.1.47`.

## Integration decisions

Three merge conflicts were reconciled:

1. `plugin/smartlead-ingest.ts` preserves both atomic transaction rollback and the current scoped-refresh failure status contract.
2. `scripts/test-host-bundle-inventory.mjs` runs both the current automatic MCP-start refresh fallback checks and the Smartlead/`all` provider-aware bundled startup regressions.
3. `docs/MCP_RESPONSE_CONTRACT.md` preserves structured selector-error coverage and replaces the obsolete unsupported-inbox-placement statement with support-gated Smart Delivery coverage.

The integration keeps Smartlead V1 read-only. No create, update, delete, stop, send, reply, forward, folder, webhook, campaign, lead, account, email, or Smart Delivery mutation was introduced.

## Public capability truthfulness

The README matrix now states:

- full core campaign/account/analytics coverage for Instantly and Smartlead;
- bounded lead/reply evidence where collection is intentionally limited;
- Instantly per-email deliverability versus Smartlead aggregate/diagnostic Smart Delivery evidence;
- Smart Delivery's separate support gate and explicit unsupported capability state;
- provider-qualified cross-provider analysis;
- the read-only product boundary.

## Release mechanism

SendLens does not publish an npm package and has no Railway service in this repo. Deployment is the tag-driven GitHub Release workflow in `.github/workflows/release.yml`. Tag `v0.1.47` must match `package.json`, pass the release gate again in GitHub Actions, and publish all four host bundles, installers, checksums, and the release manifest.
